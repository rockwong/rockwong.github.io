JS-303: Async Patterns in JavaScript

Asynchronous programming is at the heart of JavaScript — from handling user events to fetching data over the network. Over the years, the language has evolved three distinct patterns for managing async flow, and understanding when to use each one is key.

Callbacks

The original pattern. A function is passed as an argument and invoked once the operation completes. Simple for one-off tasks, but nesting them leads to the infamous callback pyramid. Still useful for event listeners and simple timeouts.

fs.readFile('data.json', (err, data) => {
  if (err) return console.error(err);
  console.log(JSON.parse(data));
});

Promises

Promises wrap an async operation in an object that will eventually resolve or reject. They're chainable with .then() and .catch(), which flattens the nesting problem. Native since ES2015 and supported everywhere that matters.

fetch('/api/posts')
  .then(res => res.json())
  .then(posts => console.log(posts))
  .catch(err => console.error(err));

Async / Await

Syntactic sugar over promises that lets you write async code that reads like synchronous code. Use async to declare a function that returns a promise, and await to pause execution until a promise settles. Error handling goes back to familiar try/catch blocks.

async function loadPosts() {
  try {
    const res = await fetch('/api/posts');
    const posts = await res.json();
    console.log(posts);
  } catch (err) {
    console.error(err);
  }
}

Which One to Choose

For new code, async/await is the default — it's the most readable and the easiest to debug. Use raw promises when you need concurrency patterns like Promise.all() or Promise.race(). Callbacks still have their place in event-driven code and older Node.js APIs, but wrap them in a promise if the nesting gets deep.
