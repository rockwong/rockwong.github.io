API-404: Designing REST Endpoints

A well-designed API feels obvious. Resources are named consistently, verbs are mapped to HTTP methods, and errors speak a language the client can parse. Here's a practical guide to getting the fundamentals right.

Resource Naming

Use plural nouns for collections and nest sub-resources logically. Keep URLs predictable — a developer who knows one endpoint should be able to guess the next.

MethodEndpointPurpose
GET/articlesList articles
GET/articles/42Get a single article
POST/articlesCreate an article
PUT/articles/42Replace an article
PATCH/articles/42Partial update
DELETE/articles/42Remove an article

Status Codes That Tell a Story

Use the right HTTP status code — not everything is a 200 or a 500. A 201 Created tells the client the resource exists now. A 422 Unprocessable Entity says the request was syntactically valid but semantically wrong. The status code is the first thing a client sees; make it count.

Error Bodies Matter

When something goes wrong, return a structured error — not a bare string. Include a machine-readable code, a human-readable message, and enough detail for the client to recover without scraping your docs.
