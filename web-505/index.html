Web-505: A Performance Primer

Web performance isn't a single metric — it's a stack of concerns spanning the network, the render pipeline, and the runtime. Small improvements at each layer compound into experiences that feel instant.

Network

The fastest request is the one you never make. Before optimizing how data arrives, ask whether it needs to arrive at all in that moment. After that, the standard playbook applies:

  • Minify and compress assets — every byte counts on a slow connection
  • Use a CDN to serve static files from the edge
  • Cache aggressively with proper Cache-Control headers
  • Inlining critical CSS avoids a render-blocking round trip

Render

Once the bytes arrive, the browser has work to do. Parse HTML, build the DOM, resolve CSS, compute layout, and paint. Keeping this pipeline smooth means:

  • Avoid layout thrashing — batch reads and writes to the DOM
  • Use will-change sparingly to hint at upcoming animations
  • Keep your selector complexity low; deeply nested selectors slow down style recalculation
  • Lazy-load images and iframes that sit below the fold

Runtime

JavaScript is the most expensive resource on the page — not in bytes, but in parsing and execution time. Ship less code. Tree-shake your bundles. Split vendor code from application code. And measure everything: a Lighthouse audit costs five minutes and can surface low-hanging fruit you've been walking past for months.

Measure First

Don't optimize based on intuition. Run a Lighthouse audit, check your Core Web Vitals, and profile with the Performance tab in DevTools. Find the bottleneck, fix it, measure again. Repeat.
